Layer 1

🧠AI — Artificial Intelligence

AI is the broadest term here. It means any system that mimics or simulates human intelligence.

  • Recognizing images (computer vision)
  • Playing chess or Go (game AI)
  • Recommending videos (recommendation engines)
  • Understanding and generating language (LLMs)
Analogy
Think of AI like "Vehicles." It's the category. Cars, trucks, motorcycles — they're all vehicles, but they work differently. LLMs are one type of AI vehicle.

Not all AI is LLMs. Spam filters and self-driving cars are AI too.

Layer 2

📖LLM — Large Language Model

An LLM is a specific type of AI — one trained to predict the next word in a sequence. Train it on enough data, and it starts to seem like it understands.

Tokens

The smallest unit of text an LLM processes. Not exactly words — can be a word, part of a word, or punctuation.

1000 tokens ≈ 750 words

Context Window

The maximum text the LLM can "hold in memory" at once — your prompt + its response combined.

Like working memory

Temperature

Controls randomness. Low (0.1) = predictable. High (0.9) = creative.

Textbook vs. Creative dial

Layer 3

🔌API — Application Programming Interface

An API is how programs talk to each other. Instead of you typing into a chat window, your code sends a message to the LLM and gets a response back.

// Your code sends a request to the LLM POST https://api.openai.com/v1/chat/completions Body: { "model": "gpt-4", "messages": [ { "role": "user", "content": "Hello!" } ] } // Response comes back { "choices": [{ "message": { "content": "Hi there!" } }] }
Analogy
An API is like a phone call. You dial a number (the endpoint), say something (the request), and get a reply (the response). The LLM is on the other end of the line.

Layer 4b — Skills

🎓Agent Skills

Skills are pre-built instruction sets that teach an agent how to behave in specific scenarios. They define the agent's personality, workflow style, and rules of engagement.

Analogy
If tools are what an agent can touch, skills are what an agent knows. Like the difference between having a wrench (tool) and knowing plumbing (skill).

📋 Custom Instructions

Define how the agent should respond — tone, format, level of detail, coding style preferences.

🎯 Task-Specific Skills

"Always write tests first", "Explain before coding", "Ask before deleting files" — behavioral rules.

📁 .clinerules / AGENTS.md

Project-specific skill files that live in your repo. The agent reads them automatically on every session.

🔄 Skill Modes

Switch between "architect mode" (plan first) and "builder mode" (code immediately) depending on the task.

# Example: .clinerules file Always explain your reasoning before writing code. Write tests before implementation (TDD). Never remove existing functionality without asking. Use TypeScript for all new files. Keep functions under 50 lines.

Essential Tool

📦Git — Version Control

Git is the foundation of modern software development. It tracks every change to your code, lets you experiment safely, and enables collaboration.

🕐 Time Machine

Every saved state is recoverable. Made a mistake? Go back to any previous version instantly.

🧪 Safe Experimentation

Branches let you try new ideas without breaking working code. Merge when ready, discard if not.

👥 Collaboration

Multiple people work on the same project simultaneously. Git merges changes and flags conflicts.

🤖 AI Agent Safety Net

When AI agents edit your code, Git lets you review, accept, or revert every change.

Analogy
Git is like Ctrl+Z on steroids. Instead of just undoing your last action, it saves every version of your project forever.
# The commands you'll use 90% of the time git init # Start tracking a project git add . # Stage all changes git commit -m "msg" # Save a checkpoint git branch feat # Create a parallel workspace git push # Upload to GitHub/GitLab git log --oneline # See history

Layer 5

🔗MCP — Model Context Protocol

MCP is a standard protocol that lets AI agents connect to external tools and data sources — file systems, databases, web browsers, and more.

Analogy
Before MCP: Every phone had its own charger. Every tool needed custom code.
After MCP: USB-C. One standard port. Plug in anything — the agent knows how to talk to it.

📁 File Systems

Read, write, search files on your machine

🗄️ Databases

Query data from SQL, NoSQL, etc.

🌐 Web Browsers

Scrape pages, automate web tasks

📦 Git Repos

Commit, branch, review, and push code

Pros

  • 10x faster boilerplate generation
  • Agents handle repetitive tasks
  • Instant test writing & debugging

  • Works while you focus on architecture

Advanced Patterns

🔀Sub-Agent Driven Development

Instead of one agent doing everything, a main agent delegates tasks to specialized sub-agents that work in parallel — each in an isolated workspace.

🤖 Main Agent (Orchestrator)

Receives your high-level task, breaks it down, dispatches sub-agents, and merges results.

⚡ Parallel Execution

Sub-agents run simultaneously on independent tasks. Frontend, backend, tests — all at once.

🌳 Git Worktrees

Each sub-agent gets its own isolated branch + working directory. No conflicts, no overwrites.

🔒 Isolation & Safety

If a sub-agent breaks something, only its worktree is affected. Main branch stays clean.

# Git worktrees — each sub-agent gets its own git worktree add ../feat-auth origin/feat-auth # Auth agent git worktree add ../feat-api origin/feat-api # API agent git worktree add ../feat-tests origin/feat-tests # Test agent # Each agent works independently in parallel # When done: review, merge, clean up git worktree remove ../feat-auth
Analogy
Single agent = one contractor doing plumbing, electrical, and painting sequentially.
Sub-agents + worktrees = a team of specialists working in parallel, each in their own room.

Best Practices

🛡️Best Practices for Using Agents & Chat

AI agents are powerful but need guardrails. These habits separate productive users from those who get burned.

🎯 Be Specific

"Add a POST /users endpoint with validation" beats "make a user API." Clear prompts = clear results.

🔍 Always Review Changes

Never blindly accept agent output. Read diffs, run tests, understand what changed before committing.

📦 Small Tasks, Not Monoliths

Break big requests into steps. "Add auth" → "Create user model" → "Add login route" → "Add middleware."

🔒 Protect Secrets

Never paste API keys, passwords, or tokens into chat. Use environment variables and .env files.

Golden Rule
Treat an AI agent like a fast but inexperienced junior developer. Give clear instructions, review their work, and never let them deploy unsupervised.

Best Practices — Context

🧠Context Management

Every agent session has a context window — a finite "memory" that fills up as you chat. When it's full, the agent starts forgetting earlier messages.

⚠️ What Happens When Context Is Full

The agent drops the oldest messages. It loses your original instructions, earlier decisions, and the reasoning behind them. Results get inconsistent or wrong.

✂️ Trim the Conversation

Start a new session for a new task. Don't let one chat grow forever. Summarize what was decided, then reset.

📌 Pin Key Information

Use AGENTS.md, .clinerules, or pinned messages to store critical info. The agent reads these every session — they don't eat chat context.

🗂️ Use Sub-Agents for Big Tasks

Instead of one long conversation, dispatch sub-agents for independent work. Each gets a fresh context window.

Analogy
Context window is like a whiteboard. When it's full, you have to erase the oldest stuff to make room. If you erase the wrong thing, the agent forgets what it was doing.
// Signs your context is overloaded → Agent repeats things you already told it → It ignores earlier instructions or constraints → Responses become generic or off-topic → It "forgets" decisions made 20+ messages ago // Fix: summarize + reset "Here's what we've decided so far: [summary]. Start a fresh session with this context."

Best Practices — Prompts

💬Effective Prompt Patterns

Do This

  • Provide context: "In this Express app, add..."
  • Specify output format: "Return TypeScript with JSDoc"
  • Set constraints: "Max 30 lines, no external deps"
  • Ask for reasoning: "Explain before you code"
  • Iterate: "Now add error handling"

Avoid This

  • Vague requests: "Fix the app"
  • Assuming correctness without testing
  • Pasting entire codebases in one prompt
  • Letting agents modify production data
  • Skipping git commits before agent sessions
// Good prompt structure Context: "Express + TypeScript REST API" Task: "Add POST /api/users with email validation" Constraints: "Use Zod for validation, max 40 lines" Output: "Route file + one test case" Rule: "Explain your approach before coding"
